Procedure for making Cloudflare the public DNS publisher for `flapjack.foo`.
This unblocks ACM certificate DNS validation, the HTTPS ALB listener, SES domain
verification, SES DKIM, and public staging health checks.

## Prerequisites

- AWS CLI credentials for the staging account.
- Cloudflare token with Zone:Read and DNS:Edit on the `flapjack.foo` zone.
- `CLOUDFLARE_API_TOKEN` and `CLOUDFLARE_ZONE_ID` available in the operator shell
  or loaded from the secrets file used for staging operations. For the
  `flapjack.foo` zone, the scripts also accept
  `CLOUDFLARE_EDIT_READ_ZONE_DNS_API_TOKEN_FLAPJACK_FOO` and
  `CLOUDFLARE_ZONE_ID_FLAPJACK_FOO`.
- Current staging AMI ID.

Verify the Cloudflare token and zone before Terraform:

```bash
export CLOUDFLARE_API_TOKEN="${CLOUDFLARE_API_TOKEN:-${CLOUDFLARE_EDIT_READ_ZONE_DNS_API_TOKEN_FLAPJACK_FOO:-}}"
export CLOUDFLARE_ZONE_ID="${CLOUDFLARE_ZONE_ID:-${CLOUDFLARE_ZONE_ID_FLAPJACK_FOO:-}}"

curl -fsS \
  -H "Authorization: Bearer ${CLOUDFLARE_API_TOKEN}" \
  -H "Content-Type: application/json" \
  "https://api.cloudflare.com/client/v4/zones/${CLOUDFLARE_ZONE_ID}"
```

Expected: `"success":true` and `"name":"flapjack.foo"`.

## 1. Verify Pages Preconditions (Required Before Terraform)

Confirm the operator token can manage Pages and confirm the marketing project is
deployed before DNS mutation:

```bash
wrangler whoami
wrangler pages project list
wrangler pages deployment list --project-name flapjack-marketing
wrangler pages domain list --project-name flapjack-marketing
```

Expected:

- `flapjack-marketing` exists.
- `flapjack-marketing.pages.dev` serves the Stage 1 `marketing/` artifact.
- Custom domains include both `flapjack.foo` and `www.flapjack.foo`.

Abort immediately if Pages project creation/deploy/domain APIs are unavailable.

## 2. Plan Cloudflare DNS Records

The Terraform `dns` module keeps AWS ownership of ACM, ALB, target groups, and
listeners. Public DNS records are `cloudflare_dns_record` resources in the same
module.

Source `api_ami_id` from the running control-plane instance `ImageId` and
`flapjack_ami_id` from `/fjcloud/<env>/aws_ami_id`; never copy one value into both.

```bash
cd ops/terraform/_shared

terraform plan \
  -var="env=staging" \
  -var="api_ami_id=ami-0123456789abcdef0" \
  -var="flapjack_ami_id=ami-0fedcba9876543210" \
  -var="domain=flapjack.foo" \
  -var="cloudflare_zone_id=${CLOUDFLARE_ZONE_ID}"
```

Review the plan carefully. Expected DNS-scope changes are Cloudflare CNAME
records for:

- `flapjack.foo`
- `api.flapjack.foo`
- `www.flapjack.foo`
- `cloud.flapjack.foo`
- ACM validation CNAMEs generated by AWS ACM
- SES DKIM CNAMEs generated by AWS SES

Do not proceed if the plan proposes unrelated ALB, RDS, EC2, VPC, IAM, or
monitoring churn.

Save the plan and review it before apply:

```bash
terraform plan \
  -target=module.dns \
  -var="env=prod" \
  -var="api_ami_id=ami-0123456789abcdef0" \
  -var="flapjack_ami_id=ami-0fedcba9876543210" \
  -var="domain=flapjack.foo" \
  -var="cloudflare_zone_id=${CLOUDFLARE_ZONE_ID}" \
  -out="../artifacts/prod-dns-cutover.tfplan"
```

Abort if plan output changes any non-public resources or mutates public records
outside `flapjack.foo` and `www.flapjack.foo`.

## 3. Apply DNS-Scope Changes

```bash
terraform apply "../artifacts/prod-dns-cutover.tfplan"
```

Cloudflare flattens the apex CNAME, so `dig` may return A records for
`flapjack.foo` even though Terraform manages it as a CNAME.

## 4. Validate ACM And Public Routing

Run the automated staging smoke harness:

```bash
bash ops/terraform/tests_stage7_runtime_smoke.sh \
  --env staging \
  --domain flapjack.foo \
  --api-ami-id ami-0123456789abcdef0 \
  --flapjack-ami-id ami-0fedcba9876543210 \
  --apply
```

The harness validates:

- Cloudflare token and zone ID.
- Cloudflare public CNAME records target an AWS ALB.
- ACM certificate status is `ISSUED`.
- ALB HTTPS listener exists.
- Target group has healthy targets.
- SES identity and DKIM are verified.
- `https://api.flapjack.foo/health` returns 200.

## 5. Verify SES Records

If SES remains blocked, inspect identity state:

```bash
aws sesv2 get-email-identity \
  --email-identity flapjack.foo \
  --region us-east-1
```

Expected: `VerificationStatus=SUCCESS` and `DkimAttributes.Status=SUCCESS`.
Terraform manages the SES domain identity and publishes the Easy DKIM CNAMEs
through Cloudflare.

Check DNS records when SES is not verified:

```bash
for token in $(aws sesv2 get-email-identity \
  --email-identity flapjack.foo \
  --region us-east-1 \
  --query 'DkimAttributes.Tokens[]' \
  --output text); do
  dig +short CNAME "${token}._domainkey.flapjack.foo" @1.1.1.1
done
```

## Rollback

Prefer reverting the Terraform change and applying the previous known-good
configuration:

```bash
terraform plan \
  -var="env=staging" \
  -var="api_ami_id=ami-0123456789abcdef0" \
  -var="flapjack_ami_id=ami-0fedcba9876543210" \
  -var="domain=flapjack.foo" \
  -var="cloudflare_zone_id=${CLOUDFLARE_ZONE_ID}"
```

For urgent DNS rollback, disable or replace only the affected
`cloudflare_dns_record` entries in Cloudflare. Keep ACM and SES validation
records in place unless the rollback intentionally disables HTTPS and email.

Record the exact command output and any blocker in `docs/runbooks/staging-evidence.md`.
