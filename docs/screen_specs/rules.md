# Rules Tab Screen Spec

## Scope

- Primary route: `/console/indexes/[name]` rules tab
- Related routes: `/console/indexes/[name]`, `/console/indexes/[name]/merchandising` (the per-query studio that consumes the same Rule resource)
- Related specs: [`_component_EditorDialog.md`](_component_EditorDialog.md), [`_component_ConfirmDialog.md`](_component_ConfirmDialog.md), [`index_detail.md`](index_detail.md)
- Audience: authenticated customers tuning relevance through query rules (promote, hide, filter, replace-query, return-userData, auto-facet-filter)
- Priority: P1 (per the [2026-05-25 parity audit](../audits/feature-parity/20260525T165423Z_fjcloud_vs_engine_dashboard_extension/SUMMARY.md) recommendation 6; depends on `EditorDialog` + `ConfirmDialog` foundation lanes)

## User Goal

Author, audit, edit, and remove ranking rules for one index — including pin/hide consequences, conditional patterns with anchoring, query rewrites, validity windows, and automatic facet filters — through a structured editor that never requires hand-writing Algolia rule JSON.

## Target Behavior

The tab shows a `Rules` panel with a header (back-link breadcrumb, `Rules` heading, count badge, `Merchandising Studio` link, `Clear All`, `Add Rule`), a search input that narrows the list as the user types, and a list of rule rows. Each row shows the enabled/disabled state icon, the `objectID`, badges for `N pinned` / `N hidden`, the user-supplied description (if any), and a one-line human-readable summary generated by a `buildRuleDescription` helper ported from upstream (e.g. `When query is "shoes", pin 3 results, hide 1 result`). Row actions are `Edit` and `Delete`. `Add Rule` and `Edit` both mount the same `EditorDialog` configured with the Rule schema; `Delete` and `Clear All` both mount `ConfirmDialog`. The JSON `<textarea>` pattern of the current implementation is explicitly disallowed in target state — the schema-driven Form view is the primary editor; a read-only `JSON Preview` panel inside the dialog mirrors the saved shape live.

## Required States

- Loading: skeleton placeholder rows render in the list area while `searchRules` resolves; header chrome (search input + `Add Rule`) is visible immediately; count badge is omitted until the response lands.
- Empty: list area shows `No rules` heading, an explanatory paragraph (`Rules let you customize search results for specific queries. Pin products to the top, hide irrelevant results, or modify queries.`), and two CTAs: `Create a Rule` (opens add-dialog) and `Open Merchandising Studio` (route navigation).
- Error: load failure shows tab-local error copy in place of the list; the header chrome stays interactive so the user can retry by editing the search input or by adding a rule.
- Populated: rule rows render as described in Target Behavior; count badge shows total hit count from the server.
- Search-filter-active: typing in the search input re-issues `searchRules` with the `query` parameter; the count badge reflects the filtered hit count; clearing the input returns to the unfiltered list.
- Search-filter-no-results: search query returns zero hits; list area shows `No rules match "<query>"` distinct from the global Empty state — the `Create a Rule` CTA is not rendered in this sub-state.
- Add-dialog: `EditorDialog` opens with `createEmptyRule()` seed (one empty condition, empty consequence, `enabled: true`, generated `objectID` of form `rule-<timestamp>`); the form body groups four expandable sub-sections — Conditions, Consequences, Validity, JSON Preview — all mounted simultaneously rather than tabbed. The JSON-preview panel updates on every keystroke.
- Add-dialog · editing-conditions: at least one condition row visible (a row with all-empty fields is allowed but counts as `no conditions configured` once normalized); per-condition validators run on blur and surface inline.
- Add-dialog · editing-consequences: at least one consequence sub-block has content (promote, hide, query-modification, parameters, userData, renderingContent, automaticFacetFilters, or automaticOptionalFacetFilters); save-disabled if normalization strips all of them.
- Add-dialog · editing-validity: zero or more time-range rows; zero ranges renders the helper text `No time restrictions — rule is always active.`
- Add-dialog · json-preview-visible: read-only `<pre>` panel mirrors `normalizeRuleForSerialization(draftRule)` and exposes a copy-to-clipboard button; never editable.
- Add-dialog · saving / save-error / dirty-cancel-confirm: per `EditorDialog` spec; field-level server errors map back into the originating sub-section (e.g. a `conditions[0].pattern is required` server error highlights the first condition's pattern field via `role="alert"`).
- Edit-dialog: same sub-states as Add-dialog, but `initialValue` is the existing rule normalized via `normalizeRule`; dialog title shows `Edit Rule: <objectID>`; saved value round-trips byte-equivalent (modulo Algolia normalization) to the original.
- Delete-confirm: `ConfirmDialog` opens with title `Delete Rule`, body `Are you sure you want to delete rule <objectID>? This action cannot be undone.` (objectID rendered in `<code>` styling), destructive variant, primary label `Delete`.
- Clear-all-confirm: `ConfirmDialog` opens with title `Delete All Rules`, body `Delete ALL rules for this index? This cannot be undone.`, destructive variant, primary label `Delete All`. The button only appears when unfiltered count > 0.
- Saving-bulk: while `clearRules` is in flight, the `Clear All` confirm's primary button shows the pending label and is disabled; the list shows a transient overlay so newly-added rules don't appear to be safe.

## Mobile Narrow Contract

Baseline viewport: 390px wide. Header chrome stacks: breadcrumb + heading + count on row 1; `Add Rule` + `Clear All` + `Merchandising Studio` collapse to icon-only buttons with `aria-label`; search input is full-width on row 2. Rule rows show `objectID` + badges on row 1 and the description + summary line wrapping below; `Edit` and `Delete` actions remain reachable. `EditorDialog` follows the component spec's mobile contract (full-width minus 16px padding, sticky header+footer, scrollable body); the compound sub-sections (Conditions / Consequences / Validity / JSON Preview) each render as collapsible blocks rather than side-by-side panes at narrow width.

## Controls And Navigation

**Tab entry:** the Rules tab lazy-mounts on first click from the index-detail tab list (see [`index_detail.md`](index_detail.md)); URL state is the tab segment of `/console/indexes/[name]` and survives a hard reload.

**Header chrome:**

- Back-link breadcrumb to `/console/indexes/[name]` with the index name as label.
- `Rules` heading and count badge (`data-testid="rules-count-badge"`) showing the server's hit-count for the current filter.
- `Merchandising Studio` link routes to `/console/indexes/[name]/merchandising` (this spec does not own that route).
- `Clear All` button (`data-testid="rules-clear-all"`, rendered only when unfiltered count > 0) opens the clear-all `ConfirmDialog`.
- `Add Rule` button (`data-testid="rules-add-button"`) seeds `createEmptyRule()` into `EditorDialog` and opens it in create mode.

**List controls:**

- Search input (`data-testid="rules-search-input"`) updates the list as the user types (debounced ~200ms; passes `query` to `searchRules`).
- Per-row `enabled` indicator (Power / PowerOff icon with `data-testid="rule-enabled-icon"` / `rule-disabled-icon`) is a state display only; toggling enabled happens inside the editor.
- Per-row `Edit` button mounts the editor in edit mode for that rule's `objectID`; trigger element is recorded so focus returns there on dialog close.
- Per-row `Delete` button (`aria-label="Delete rule <objectID>"`) opens the delete `ConfirmDialog`.

**Editor schema** (consumed by `EditorDialog`'s field-schema prop):

- `objectID` — text field, required, validates non-empty after trim.
- `description` — text field, optional.
- `enabled` — toggle field, default `true`.
- `conditions` — array field, `addLabel: "Add Condition"`, helper text `Leave empty for a conditionless rule that always applies.` Each item is a compound block:
  - `pattern` — text field, optional alone but required if `anchoring` set.
  - `anchoring` — select field with options `[is, startsWith, endsWith, contains]`, required if `pattern` set; cross-field validator surfaces `Condition N: anchoring is required when pattern is provided.` and the reverse.
  - `alternatives` — toggle field, default false.
  - `context` — text field, optional (must serialize alongside the rest only if non-empty).
  - `filters` — text field, optional.
- `consequence` — compound block, the dialog's largest sub-section. Order matches upstream:
  - Promoted Items — array of `{objectID: text required, position: number min=0}`, `addLabel: "Add Promoted Item"`. Duplicate-objectID validator surfaces `Duplicate objectID in promoted items.`
  - Hidden Items — array of `{objectID: text required}`, `addLabel: "Add Hidden Item"`.
  - Query Modification — radio with three modes: `none` (clears `consequence.params.query`), `literal` (single text input for the replacement query), `edits` (array of `{type: select [remove, replace], delete: text required, insert: text required when type=replace}`).
  - Parameters — `filters` (text), `hitsPerPage` (number min=0), `aroundLatLng` (text), `aroundRadius` (text accepting `all` or number), `restrictSearchableAttributes` (text comma-list), `facetFilters` / `numericFilters` / `optionalFilters` / `tagFilters` (JSON textareas), `filterPromotes` (toggle).
  - User Data — JSON textarea; if non-empty, must parse — `Invalid JSON in User Data field.` otherwise.
  - Rendering Content — JSON textarea with same validation.
  - Automatic Facet Filters — array of `{facet: text required, disjunctive: toggle, score: number}`.
  - Automatic Optional Facet Filters — array of same shape.
- `validity` — array field, `addLabel: "Add Time Range"`, helper text `No time restrictions — rule is always active.` when empty. Each item is `{from: datetime-local, until: datetime-local}` stored as unix seconds; cross-field validator `Valid Until must be after Valid From`.

**JSON Preview panel:**

- Rendered inside the dialog body below the form sub-sections (`data-testid="rule-json-preview"`).
- Reflects `normalizeRuleForSerialization(draftRule)` byte-equivalent to what `saveRule` will receive.
- Read-only `<pre>` with a copy-to-clipboard button.
- Explicitly **no** Form/JSON tab switcher, **no** JSON textarea-as-editor, and **no** Monaco editor — upstream's `lazy(() => import('@monaco-editor/react'))` pattern is not ported.

**Dialog mount and focus:**

- `Add Rule` / per-row `Edit` / per-row `Delete` / `Clear All` all record their trigger element as `lastTriggerEl` before opening the dialog.
- On dialog close (any path: save success, cancel, confirm dismiss), focus returns to `lastTriggerEl` per `EditorDialog` / `ConfirmDialog` component contracts.
- Dialog mount uses Svelte portal/teleport so the modal is sibling to `<body>`, never inside the tab panel (prevents stacking-context issues with the index-detail header).

## Acceptance Criteria

- [ ] Given an index with three seeded rules, when the user opens the Rules tab, then the count badge shows `3` and three rule rows render with their `objectID`, enabled indicator, and `buildRuleDescription` summary text (not raw JSON).
- [ ] Given the populated tab, when the user clicks `Add Rule`, then `EditorDialog` opens in create mode with an empty condition row, empty consequence sections, `enabled` toggle on, and a generated `objectID` of the form `rule-<timestamp>` in the editable Object ID field.
- [ ] Given the add-dialog open, when the user enters a pattern but leaves anchoring unselected, then on Save the per-field validator surfaces `Condition 1: anchoring is required when pattern is provided.` and Save remains disabled until the user picks an anchoring value.
- [ ] Given the add-dialog open, when the user adds a Promoted Item with `objectID="sku-123"` and `position=0`, then the JSON Preview pane immediately renders `"consequence": { "promote": [{ "objectID": "sku-123", "position": 0 }] }` (byte-equivalent to `normalizeRuleForSerialization` output).
- [ ] Given the add-dialog open, when the user adds a validity range with `from` set to 2026-01-01T00:00 and `until` left at 0, then the JSON Preview shows the range under `"validity"` with the `from` unix timestamp and Save is disabled with a per-field error indicating `until` must be after `from`.
- [ ] Given the add-dialog open with a valid rule, when the user clicks Save, then `saveRule(indexName, objectID, normalizedRule)` is invoked, the dialog closes on resolution, a success toast appears, and the new row appears in the list with its `buildRuleDescription` summary.
- [ ] Given an existing rule with a literal query-replacement consequence (`consequence.params.query="trousers"`), when the user opens its Edit dialog, makes no changes, and clicks Save, then the persisted rule round-trips byte-equivalent (after Algolia normalization) to the original.
- [ ] Given a populated list with rules whose `objectID` matches `boost-*` and `hide-*`, when the user types `boost` in the search input, then only the `boost-*` rows remain visible and the count badge reflects the filtered count.
- [ ] Given a rule row, when the user clicks Delete, then `ConfirmDialog` opens with the rule's `objectID` rendered in the body; clicking Cancel closes the confirm without invoking `deleteRule`; clicking Delete invokes `deleteRule(indexName, objectID)`, closes the confirm, and removes the row.
- [ ] Given count > 0 and the user clicks `Clear All`, then `ConfirmDialog` opens with the destructive `Delete All` label; clicking Delete All invokes `clearRules(indexName)` (a new API client method), closes the confirm, and the list transitions to the Empty state.
- [ ] Given the editor has unsaved changes, when the user clicks Cancel/X/Esc/backdrop, then the dirty-cancel-confirm pattern from `EditorDialog` engages (per that component's spec), preventing accidental loss of an in-progress rule.
- [ ] Given the server rejects a save with a per-field error map (e.g. `{ "consequence.promote[0].objectID": "objectID not found in index" }`), when the rejection settles, then the dialog stays open, the Promoted Items section shows the field-level error in `role="alert"`, and Save is disabled until the offending field is edited.
- [ ] Given the list is in the empty state, when the user clicks `Create a Rule`, then the same add-dialog flow above runs and the empty state transitions to populated on save.
- [ ] Given a rule with `enabled: false`, when the row renders, then the `rule-disabled-icon` testid is present (not the `-enabled-icon`); and given the user toggles `enabled` to `false` in the editor and saves, then the row re-renders showing the disabled icon without a page reload.
- [ ] Given the add-dialog open with two Promoted Items and one Hidden Item, when the user clicks the second Promoted Item's `Remove Promote 2` button, then only that item is removed, the first Promoted Item is preserved, the Hidden Item is preserved, and the JSON Preview updates to reflect the new shape.

## Current Implementation Gaps

- Shipped: `web/src/routes/console/indexes/[name]/tabs/RulesTab.svelte` is 149 lines and consists of a flat HTML table plus an always-visible `<textarea>` that requires users to hand-write the Algolia rule JSON literal. There is no editor dialog, no condition/consequence/validity editor, no JSON-preview pane, no `Clear All`, no count badge, no search input, no per-row Edit button, no destructive-action confirmation, and no `buildRuleDescription` summary — rows show `description ?? '-'` only. The Rules tab is the worst-parity tab in the audit at 1 present / 5 partial / 13 absent of 19 catalog rows.
- Trust calibration: the parent audit at `../audits/feature-parity/20260524T174411Z_fjcloud_vs_engine_dashboard/SUMMARY.md:146-162` graded 13 Rules rows as `present` that were actually `absent` or `partial`; this extension's [tab_rules.md](../audits/feature-parity/20260525T165423Z_fjcloud_vs_engine_dashboard_extension/tab_rules.md) supersedes those verdicts.
- Backend dependency: `web/src/lib/api/client.ts` exposes `searchRules`, `saveRule`, `getRule`, `deleteRule` (lines 484-507) but **no `clearRules` method**; the `Clear All` flow requires adding this method (POST `/indexes/<name>/rules/clear` per Algolia parity) before the corresponding API server route can be wired. Confirm the backend route exists before the implementation lane commits to the schema.
- Helpers dependency: upstream's `ruleHelpers.ts` (at `flapjack_dev/engine/dashboard/src/lib/ruleHelpers.ts`) must be ported to a Svelte-compatible TypeScript module at `web/src/lib/rules/ruleHelpers.ts` exporting `normalizeRule`, `createEmptyRule`, `buildRuleDescription`, `normalizeRuleForSerialization`, `validateRule`, and `prepareRuleEditorSave`. The associated test file (`ruleHelpers.test.ts`) should port too — the helper has nontrivial branching that the upstream test already covers.
- Component dependencies: this tab cannot be built until the foundation lanes for `EditorDialog` (Theme A) and `ConfirmDialog` (Theme B) land; sequencing per audit recommendations 2 and 3.

## Edge Cases

- Rule with no consequences (empty `consequence` object after stripping empty sub-objects): editor blocks Save with `Rule must have at least one consequence` and the form-level alert names which consequence sections are eligible.
- Validity range with `until <= from` (any range, including a zero `until` when `from` is set): per-field validator on the affected range surfaces `Valid Until must be after Valid From`; the `JSON Preview` still shows the in-progress shape so the user can see what they're editing.
- Very long condition pattern (e.g. > 200 chars): renders truncated with ellipsis in the row's summary line; the editor's text input does not truncate and shows the full value.
- Duplicate `objectID` in promoted items: per-field validator surfaces `Duplicate objectID in promoted items.` (mirrors the upstream `validateRule` error message).
- Invalid JSON pasted into `userData` or `renderingContent` textarea: per-field validator surfaces `Invalid JSON in User Data field.` / `Invalid JSON in Rendering Content field.`; Save is disabled until the JSON parses.
- Server returns a rule whose `objectID` differs from the path segment used at save (legacy data drift): edit-dialog title shows the server-returned `objectID`; on Save, the editor uses the in-form `objectID` as the path parameter, allowing rename. The implementation lane should confirm the backend supports rename-via-PUT and add a known-answer test that verifies the old path 404s after rename.
- Concurrent edit by another user: out of scope (no optimistic locking); a stale-write rejection surfaces via the `EditorDialog` Save-error path with the server's message.
- Search input with a query matching zero rules: list renders an empty-search state (`No rules match "<query>"`) distinct from the global Empty state — the Empty state's `Create a Rule` CTA only renders when the unfiltered count is zero.
- Search input debounce vs Save race: the user types fast enough that a search response arrives while an edit dialog is open for a row that the new search filters out; the dialog stays open (it is decoupled from the list), the row disappears from the underlying list, and on Save success the row re-appears (or appears for the first time) only if it matches the current search query — never silently saved and lost.
- Rule with `consequence.userData` already-parsed object (server-returned), edit dialog must serialize back to JSON for the textarea; on save the helper re-parses it before sending, so the round-trip is `string ↔ object`, never `string ↔ string` (which would double-stringify on subsequent saves).
- Browser back/forward while a dialog is open: navigating away triggers the dirty-cancel-confirm if the form is dirty; if the user discards, the route change proceeds; if they keep editing, the route change is blocked via SvelteKit's `beforeNavigate` until they explicitly close the dialog.
